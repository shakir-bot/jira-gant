# Гант + Jira — локальное приложение

Диаграмма Ганта с прямой авторизацией в Jira Cloud. Работает у вас на компьютере:
сервер запускается локально, обращается к Jira напрямую с вашей машины,
никакие данные никуда наружу (кроме самой Jira) не уходят.

## Что умеет

- Вход через веб-форму по email + персональному API-токену Jira (это официальный,
  поддерживаемый Atlassian способ авторизации для личных скриптов/инструментов —
  без пароля, с возможностью в любой момент отозвать токен в настройках Jira).
- Выбор любого сохранённого фильтра Jira (избранные показываются сразу, остальные — по поиску).
- Диаграмма Ганта: перетаскивание задач по датам, изменение длительности, связи между
  задачами, вехи, ручное добавление задач, перетаскивание задач из бэклога на график,
  drag-переупорядочивание, группировка по системе/приоритету, масштаб неделя/месяц/квартал,
  вид «Календарь».
- Редактирование **статуса** и **исполнителя** прямо в диаграмме — изменения пытаются
  синхронизироваться с Jira через API (см. ограничения ниже).

## Запуск

Требуется Node.js версии 18 или новее (проверить: `node --version`).

```bash
cd gantt-jira-app
npm install
npm start
```

Откройте в браузере: **http://localhost:3210**

## Вход

1. **Адрес Jira** — например `detmir.atlassian.net` (без https://, можно и с ним).
2. **Email** — тот, под которым вы входите в Jira.
3. **API-токен** — создаётся на странице
   https://id.atlassian.com/manage-profile/security/api-tokens
   (кнопка «Create API token»). Токен действует как ваш аккаунт — храните его
   так же, как пароль, и отзывайте, если он больше не нужен.

Сервер проверяет токен прямым запросом к Jira (`/rest/api/3/myself`) и по
умолчанию хранит его только в оперативной памяти процесса на время сессии
(8 часов или до перезапуска сервера) — на диск ничего не пишется.

Если поставить галочку **«Запомнить на этом устройстве»** при входе, данные
для входа (адрес, email, токен) сохранятся в файле `.jira-config.json` рядом
с сервером (он в `.gitignore` — не попадёт в git, права доступа к файлу
ограничены только вашим пользователем ОС). При следующем запуске сервера или
после истечения сессии не нужно будет вводить их заново — приложение само
подставит сохранённые данные. Кнопка «Выйти» стирает и активную сессию, и
этот сохранённый файл.

## Что сохраняется, а что синхронизируется с Jira

- **Даты (перетаскивание/изменение длительности), связи между задачами, вехи,
  вручную добавленные задачи, ручной порядок строк** — сохраняются только
  локально в браузере (localStorage), отдельно для каждого фильтра. В саму
  Jira не пишутся (там нет места для «связей» и «вех» в нашем понимании, а
  плановые даты — вопрос отдельной договорённости о полях).
- **Статус и исполнитель** — при редактировании прямо в диаграмме приложение
  пытается применить изменение в реальной задаче Jira через API. Внизу справа
  появится зелёное подтверждение или красное предупреждение, если изменение
  не прошло (например, у вас нет прав на этот переход статуса в workflow).
  Изменение всё равно останется в интерфейсе локально, даже если синхронизация
  не удалась — но в Jira эта задача будет отличаться, пока вы не поправите
  вручную.
- Кнопка **«🔄 Обновить из Jira»** перезагружает свежие данные по фильтру и
  стирает локальные правки для него (после подтверждения) — полезно, если
  кто-то другой изменил задачи в самой Jira.

## Настройка полей (если диаграмма не под CAB-2177/ЭТрН)

Даты начала/окончания и счётчик переносов берутся из конкретных полей Jira,
специфичных для проекта, под который это делалось изначально. Если нужно
собрать диаграмму по другому проекту с другими полями — откройте
`server.js` и поправьте константы в начале файла:

```js
const FIELD_START = 'customfield_10248';       // Дата начала
const FIELD_PLAN_END = 'customfield_10311';    // Плановая дата завершения
const FIELD_RESCHEDULE = 'customfield_10537';  // Количество переносов плановой даты
```

Узнать ID нужных полей можно в Jira: Настройки → Поля, либо через запрос
`GET /rest/api/3/field` (вернёт список всех полей с их ID и названиями).

Если у задач нет обеих дат (начала и окончания) — они попадают в таблицу
«Бэклог» внизу диаграммы вместо самого графика; их можно перетащить на график
мышкой, указав даты вручную.

## Ограничения этой версии

- Это **не** полноценный вход «Log in with Jira» через OAuth-экран Atlassian
  (с редиректом на страницу авторизации Atlassian и кнопкой «Разрешить
  доступ»). Такой вариант тоже возможен, но требует регистрации отдельного
  OAuth-приложения в Atlassian Developer Console (Client ID/Secret) — это
  занимает 10–15 минут в вашем аккаунте Atlassian. Если хотите такой вариант
  вместо API-токена — дайте знать, дополню.
- Работает только с Jira **Cloud** (`*.atlassian.net`). Jira Server/Data
  Center использует другой протокол авторизации и здесь не поддерживается.
- Сессии хранятся в памяти процесса сервера (`express-session`, `MemoryStore`)
  — при перезапуске сервера все активные входы слетают (тем, кто поставил
  «запомнить на этом устройстве», не нужно будет вводить данные заново).
  Для пары-тройки одновременных пользователей этого достаточно; для крупной
  команды на постоянной основе стоит подключить внешнее хранилище сессий
  (Redis и т.п.) — сейчас это не сделано.

## Работа несколькими людьми / запуск на другом компьютере в сети

Сервер можно запустить один раз на одном компьютере, а открывать его с других
компьютеров в той же локальной сети (офис, домашний Wi-Fi) — не обязательно
поднимать инстанс на каждой машине отдельно.

1. Запустите `npm start` на компьютере, который будет сервером. В консоли
   при старте выводится адрес вида `http://192.168.x.x:3210` — это адрес
   для входа с других компьютеров той же сети.
2. На других компьютерах откройте этот адрес в браузере вместо `localhost`.
3. Каждый входит под своими email + API-токеном Jira — учётные данные и
   сессия отдельного человека не пересекаются с другими: авторизация Jira
   держится в серверной сессии по cookie браузера, а «запомнить на этом
   устройстве» сохраняется по отдельной записи на каждый браузер/компьютер
   (в общем файле `.jira-config.json`, но с отдельным ключом на устройство,
   так что вход одного человека не подменяет и не стирает вход другого).
4. Правки на диаграмме (перетаскивание дат, связи, вехи, состав команды)
   хранятся в `localStorage` браузера — они у каждого свои и не видны
   другим, даже если все смотрят один и тот же фильтр Jira.
5. Если с другого компьютера адрес не открывается — скорее всего, файрвол
   компьютера-сервера блокирует входящие соединения на порт 3210 (в
   macOS: Системные настройки → Сеть → Файрвол → разрешить для `node`).
